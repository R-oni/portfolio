🔄 3. ETL & Data Warehouse — Coração do Cargo Essencial

A vaga destaca ETL e Data Warehouse como responsabilidades centrais. Esse conhecimento é o que diferencia um Especialista de um Analista de BI.

3.1 O que é ETL

ETL = Extract, Transform, Load — o processo de mover dados de sistemas-fonte para o data warehouse.

EtapaO que fazExemplo no hotel
Extract (Extrair)Puxa dados dos sistemas-fonteLer reservas do PMS, vendas do PDV, dados do ERP financeiro
Transform (Transformar)Limpa, padroniza, enriquece, validaPadronizar nomes de canais, calcular noites, converter moedas, tratar NULLs
Load (Carregar)Insere no destino (DW/data mart)Carregar fato_reservas e dimensões no data warehouse

ELT vs. ETL

  • ETL — transforma ANTES de carregar (tradicional). Ferramentas: SSIS, Pentaho, Talend
  • ELT — carrega bruto, transforma NO destino (moderno). Ferramentas: dbt, SQL dentro do DW

Na prática, muitas empresas usam uma mistura. O importante é saber: de onde vêm os dados, o que precisa ser limpo/transformado, e onde eles são armazenados para consumo.

Exemplo Python/Pandas — ETL simplificado

import pandas as pd from sqlalchemy import create_engine # ── EXTRACT ── engine_pms = create_engine('postgresql://user:pass@pms-server/hotel_db') df_reservas = pd.read_sql('SELECT * FROM reservas WHERE checkin >= %s', engine_pms, params=['2026-01-01']) # ── TRANSFORM ── df_reservas['noites'] = (df_reservas['checkout'] - df_reservas['checkin']).dt.days df_reservas['noites'] = df_reservas['noites'].clip(lower=1) df_reservas['diaria_media'] = df_reservas['valor_total'] / df_reservas['noites'] df_reservas['canal'] = df_reservas['canal'].str.strip().str.title().fillna('Direto') df_reservas = df_reservas.drop_duplicates(subset=['reserva_id']) # Validação anomalias = df_reservas[df_reservas['diaria_media'] > 5000] if len(anomalias) > 0: print(f'⚠️ {len(anomalias)} reservas com diária acima de R$5000') # ── LOAD ── engine_dw = create_engine('postgresql://user:pass@dw-server/data_warehouse') df_reservas.to_sql('fato_reservas', engine_dw, if_exists='append', index=False)

3.4 Data Warehouse — Conceitos Fundamentais

Um Data Warehouse (DW) é um repositório central de dados organizados para análise e relatórios. Diferente de um banco transacional (OLTP).

AspectoOLTP (PMS, ERP)OLAP (Data Warehouse)
PropósitoOperação do dia-a-diaAnálise e relatórios
QueriesINSERT, UPDATE rápidosSELECT complexos, agregações
ModeloNormalizado (3NF)Dimensional (Star/Snowflake)
UsuáriosRecepcionistas, caixaGestores, controladoria, BI
DadosAtuais (últimas transações)Históricos (anos de dados)
PerformanceOtimizado para escritasOtimizado para leituras

Camadas do Data Warehouse

┌─────────────────────────────────────────────────────┐ │ CONSUMO │ │ Dashboards, Relatórios, Ad-hoc │ ├─────────────────────────────────────────────────────┤ │ CAMADA GOLD / CURATED │ │ Tabelas fato + dimensões (Star Schema) │ │ KPIs calculados, prontos para consumo │ ├─────────────────────────────────────────────────────┤ │ CAMADA SILVER / CLEANED │ │ Dados limpos, padronizados, validados │ │ Transformações de negócio aplicadas │ ├─────────────────────────────────────────────────────┤ │ CAMADA BRONZE / RAW │ │ Dados brutos, como vieram da fonte │ │ Histórico completo, sem alteração │ ├─────────────────────────────────────────────────────┤ │ FONTES (PMS, PDV, ERP, CRM) │ └─────────────────────────────────────────────────────┘

Na entrevista: "Minha abordagem para construir um DW é usar camadas — bronze para dados brutos, silver para dados limpos e transformados, e gold para o modelo dimensional pronto para consumo. Isso facilita a rastreabilidade e o debugging quando algo não bate."

3.5 Slowly Changing Dimensions (SCD)

Quando dados de dimensões mudam ao longo do tempo (ex: hóspede muda de cidade, quarto muda de categoria):

TipoO que fazQuando usar
SCD Tipo 1Sobrescreve o valor antigoQuando histórico não importa (ex: corrigir typo no nome)
SCD Tipo 2Cria nova linha com data de vigênciaQuando precisa do histórico (ex: mudança de tarifa do quarto)
SCD Tipo 3Adiciona coluna para valor anteriorQuando só precisa do valor atual e último anterior

Exemplo SCD Tipo 2 — Tarifa do quarto mudou

-- dim_quarto com SCD Tipo 2 CREATE TABLE dim_quarto ( sk_quarto SERIAL PRIMARY KEY, -- surrogate key nk_quarto INT, -- natural key (id original) numero VARCHAR(10), tipo VARCHAR(50), diaria_rack DECIMAL(10,2), data_inicio DATE, data_fim DATE, -- NULL = registro atual flag_atual BOOLEAN DEFAULT TRUE );

3.7 Qualidade de dados — O que a vaga pede

A vaga menciona "assegurando que os dados sejam precisos e consistentes". Isso é data quality:

Dimensões de qualidade de dados

  • Completude — há NULLs onde não deveria? (ex: reserva sem valor)
  • Unicidade — há duplicatas? (ex: mesma reserva registrada 2x)
  • Consistência — dados batem entre sistemas? (receita no PMS = receita no financeiro?)
  • Validade — valores fazem sentido? (diária de R$ -50? checkout antes do checkin?)
  • Atualidade — dados estão atualizados? (ETL rodou hoje?)

Checks de validação em SQL

-- Verificar reservas sem valor SELECT COUNT(*) AS reservas_sem_valor FROM fato_reservas WHERE valor_total IS NULL OR valor_total = 0; -- Verificar duplicatas SELECT reserva_id, COUNT(*) AS duplicatas FROM fato_reservas GROUP BY reserva_id HAVING COUNT(*) > 1; -- Verificar checkout antes do checkin SELECT * FROM fato_reservas WHERE checkout <= checkin; -- Comparar totais entre fonte e DW SELECT (SELECT SUM(valor_total) FROM pms.reservas WHERE checkin >= '2026-04-01') AS total_fonte, (SELECT SUM(valor_total) FROM dw.fato_reservas WHERE checkin >= '2026-04-01') AS total_dw;

Regra de ouro: nunca publique um dashboard sem antes validar os totais contra a fonte. Na controladoria, um número errado pode gerar decisões erradas. Sempre tenha queries de validação rodando automaticamente.

4.2 SELECTs Básicos — O Dia a Dia

Listar reservas do mês atual

SELECT r.id, h.nome AS hospede, r.checkin, r.checkout, r.valor_total, r.status FROM reservas r JOIN hospedes h ON h.id = r.hospede_id WHERE r.checkin BETWEEN '2026-04-01' AND '2026-04-30' ORDER BY r.checkin;

Quartos disponíveis para uma data

SELECT q.numero, q.tipo, q.andar FROM quartos q WHERE q.id NOT IN ( SELECT r.quarto_id FROM reservas r WHERE r.status = 'confirmada' AND r.checkin <= '2026-04-15' AND r.checkout > '2026-04-15' ) ORDER BY q.tipo, q.numero;

Hóspedes que mais vieram (frequência)

SELECT h.nome, h.email, COUNT(r.id) AS total_reservas, SUM(r.valor_total) AS receita_total, MAX(r.checkout) AS ultima_estadia FROM hospedes h JOIN reservas r ON r.hospede_id = h.id WHERE r.status = 'finalizada' GROUP BY h.id, h.nome, h.email HAVING COUNT(r.id) >= 3 ORDER BY total_reservas DESC;

4.3 KPIs Hoteleiros em SQL — O que vão te perguntar

Taxa de Ocupação (%)

Fórmula: (Quartos Vendidos / Quartos Disponíveis) × 100

-- Taxa de ocupação diária SELECT d.data, COUNT(DISTINCT r.quarto_id) AS quartos_ocupados, (SELECT COUNT(*) FROM quartos WHERE status = 'ativo') AS total_quartos, ROUND( COUNT(DISTINCT r.quarto_id) * 100.0 / (SELECT COUNT(*) FROM quartos WHERE status = 'ativo'), 1 ) AS taxa_ocupacao FROM calendario d LEFT JOIN reservas r ON d.data BETWEEN r.checkin AND r.checkout - INTERVAL '1 day' AND r.status IN ('confirmada', 'finalizada') WHERE d.data BETWEEN '2026-04-01' AND '2026-04-30' GROUP BY d.data ORDER BY d.data;

ADR — Average Daily Rate (Diária Média)

Fórmula: Receita de Hospedagem / Quartos Vendidos

-- ADR mensal SELECT DATE_TRUNC('month', r.checkin) AS mes, ROUND(SUM(r.valor_total) / SUM( DATE_PART('day', r.checkout - r.checkin) ), 2) AS adr FROM reservas r WHERE r.status = 'finalizada' AND r.checkin >= '2026-01-01' GROUP BY DATE_TRUNC('month', r.checkin) ORDER BY mes;

RevPAR — Revenue Per Available Room

Fórmula: Receita Total de Quartos / Total de Quartos Disponíveis. Ou: ADR × Taxa de Ocupação

-- RevPAR mensal WITH receita_mensal AS ( SELECT DATE_TRUNC('month', r.checkin) AS mes, SUM(r.valor_total) AS receita_hospedagem FROM reservas r WHERE r.status = 'finalizada' GROUP BY 1 ), quartos_disp AS ( SELECT COUNT(*) * 30 AS quartos_noite_disponiveis -- simplificado FROM quartos WHERE status = 'ativo' ) SELECT rm.mes, rm.receita_hospedagem, ROUND(rm.receita_hospedagem / qd.quartos_noite_disponiveis, 2) AS revpar FROM receita_mensal rm, quartos_disp qd ORDER BY rm.mes;

TRevPAR — Total Revenue Per Available Room

Como o RevPAR, mas inclui TODA receita (A&B, spa, loja, etc). Indicador mais completo.

SELECT DATE_TRUNC('month', rec.data) AS mes, ROUND(SUM(rec.valor) / (SELECT COUNT(*) * 30 FROM quartos WHERE status = 'ativo'), 2) AS trevpar FROM receitas rec GROUP BY 1 ORDER BY mes;

Se te perguntarem a diferença entre RevPAR e TRevPAR: "RevPAR mede só receita de quartos por quarto disponível. TRevPAR inclui TODA a receita — é mais útil em resorts como a Enjoy que tem restaurante, spa, loja e bar, onde a receita vai muito além da diária."

4.4 Análises que a Controladoria pede

Receita por canal de venda

SELECT c.nome AS canal, COUNT(r.id) AS qtd_reservas, SUM(r.valor_total) AS receita_bruta, SUM(r.valor_total * c.comissao_pct / 100) AS comissao, SUM(r.valor_total * (1 - c.comissao_pct / 100)) AS receita_liquida, ROUND(AVG(r.valor_total), 2) AS ticket_medio FROM reservas r JOIN canais c ON c.id = r.canal_id WHERE r.status = 'finalizada' AND r.checkin >= '2026-01-01' GROUP BY c.nome ORDER BY receita_liquida DESC;

Por que isso importa: Booking cobra ~15-20% de comissão. Reserva direta = 0%. A controladoria quer AUMENTAR venda direta e reduzir dependência de OTAs (Online Travel Agencies). Esse é um dos relatórios mais pedidos.

Budget vs. Realizado

SELECT b.departamento, b.mes, b.valor_orcado, COALESCE(SUM(d.valor), 0) AS valor_realizado, b.valor_orcado - COALESCE(SUM(d.valor), 0) AS variacao, ROUND( (COALESCE(SUM(d.valor), 0) - b.valor_orcado) * 100.0 / b.valor_orcado, 1 ) AS variacao_pct FROM budget b LEFT JOIN despesas d ON d.departamento = b.departamento AND DATE_TRUNC('month', d.data) = b.mes GROUP BY b.departamento, b.mes, b.valor_orcado ORDER BY b.mes, b.departamento;

Análise de sazonalidade

SELECT EXTRACT(MONTH FROM r.checkin) AS mes, TO_CHAR(r.checkin, 'Month') AS nome_mes, COUNT(*) AS reservas, ROUND(AVG(r.valor_total), 2) AS ticket_medio, ROUND(AVG(DATE_PART('day', r.checkout - r.checkin)), 1) AS estadia_media_dias FROM reservas r WHERE r.status = 'finalizada' GROUP BY 1, 2 ORDER BY 1;

Olímpia: Alta temporada = férias escolares (Jan, Jul, Dez) + feriados prolongados. Festa do Peão de Barretos (agosto) também traz bastante movimento. Baixa temporada = março, maio, agosto (fora Barretos). Entender essa sazonalidade é crucial.

Receita por departamento (P&L simplificado)

SELECT rec.categoria, SUM(CASE WHEN EXTRACT(MONTH FROM rec.data) = 1 THEN rec.valor ELSE 0 END) AS jan, SUM(CASE WHEN EXTRACT(MONTH FROM rec.data) = 2 THEN rec.valor ELSE 0 END) AS fev, SUM(CASE WHEN EXTRACT(MONTH FROM rec.data) = 3 THEN rec.valor ELSE 0 END) AS mar, -- ... até dezembro SUM(rec.valor) AS total_ano FROM receitas rec WHERE EXTRACT(YEAR FROM rec.data) = 2026 GROUP BY rec.categoria ORDER BY total_ano DESC;

4.5 JOINs e Subqueries mais usados

Reservas com diária abaixo da média (desconto excessivo?)

SELECT r.id, h.nome, r.valor_total, DATE_PART('day', r.checkout - r.checkin) AS noites, ROUND(r.valor_total / DATE_PART('day', r.checkout - r.checkin), 2) AS diaria_paga, tq.diaria_base AS diaria_tabela, ROUND( (tq.diaria_base - r.valor_total / DATE_PART('day', r.checkout - r.checkin)) * 100.0 / tq.diaria_base, 1 ) AS desconto_pct FROM reservas r JOIN hospedes h ON h.id = r.hospede_id JOIN quartos q ON q.id = r.quarto_id JOIN tipos_quarto tq ON tq.id = q.tipo_id WHERE r.valor_total / DATE_PART('day', r.checkout - r.checkin) < tq.diaria_base * 0.7 ORDER BY desconto_pct DESC;

Ranking de tipos de quarto por receita (Window Function)

SELECT tq.nome AS tipo_quarto, COUNT(r.id) AS reservas, SUM(r.valor_total) AS receita, ROUND(AVG(r.valor_total), 2) AS ticket_medio, RANK() OVER (ORDER BY SUM(r.valor_total) DESC) AS ranking FROM reservas r JOIN quartos q ON q.id = r.quarto_id JOIN tipos_quarto tq ON tq.id = q.tipo_id WHERE r.status = 'finalizada' GROUP BY tq.nome;

Comparação mês atual vs. mês anterior (LAG)

WITH receita_mensal AS ( SELECT DATE_TRUNC('month', data) AS mes, SUM(valor) AS receita FROM receitas GROUP BY 1 ) SELECT mes, receita, LAG(receita) OVER (ORDER BY mes) AS receita_anterior, ROUND( (receita - LAG(receita) OVER (ORDER BY mes)) * 100.0 / LAG(receita) OVER (ORDER BY mes), 1 ) AS variacao_pct FROM receita_mensal ORDER BY mes;

Análise de cancelamentos

SELECT c.nome AS canal, COUNT(*) FILTER (WHERE r.status = 'cancelada') AS canceladas, COUNT(*) AS total, ROUND( COUNT(*) FILTER (WHERE r.status = 'cancelada') * 100.0 / COUNT(*), 1 ) AS taxa_cancelamento FROM reservas r JOIN canais c ON c.id = r.canal_id GROUP BY c.nome ORDER BY taxa_cancelamento DESC;

CMV — Custo da Mercadoria Vendida (A&B)

Tabela Fato vs. Dimensão

  • Fato — contém métricas/medidas numéricas + chaves estrangeiras. Ex: fato_reservas (valor_total, noites, custos)
  • Dimensão — contém atributos descritivos para filtrar/agrupar. Ex: dim_tempo, dim_hospede, dim_quarto
  • Surrogate Key (SK) — chave artificial (SERIAL) na dimensão, diferente da chave natural do sistema fonte
  • Grain (Granularidade) — nível de detalhe de uma linha da fato. Ex: "uma linha por reserva" ou "uma linha por noite por reserva"

Na entrevista, se perguntarem sobre modelagem: "Para um hotel, eu criaria um star schema com fato_reservas como tabela central com granularidade por reserva, conectada às dimensões de tempo, hóspede, quarto e canal. Também criaria uma fato_receitas separada com granularidade por transação, para capturar A&B, spa e loja. A dim_tempo é essencial — permite análise por dia da semana, feriado, temporada."

Sempre tenha uma tabela calendário (dim_tempo). É a primeira coisa que um profissional de BI cria. Ela permite análises por dia da semana, feriado, comparação ano a ano, etc. Todas as ferramentas (Power BI, Tableau, Qlik) precisam dela.

Perguntas técnicas

"Qual a diferença entre INNER JOIN e LEFT JOIN?"

INNER JOIN retorna só linhas que existem nas duas tabelas. LEFT JOIN retorna TODAS as linhas da tabela da esquerda, mesmo que não tenha correspondência na direita (nesse caso, campos da direita vêm como NULL).

Exemplo prático: "Se faço LEFT JOIN de quartos com reservas, vejo todos os quartos — inclusive os que não têm nenhuma reserva. Com INNER JOIN, quartos sem reserva desaparecem."

"O que é GROUP BY e HAVING?"

GROUP BY agrupa linhas para usar funções de agregação (SUM, COUNT, AVG). HAVING filtra os grupos depois da agregação — é o WHERE do GROUP BY.

Exemplo: "GROUP BY canal me dá a receita por canal. HAVING SUM(valor) > 100000 filtra só canais com receita acima de 100 mil."

"O que é uma CTE (Common Table Expression)?"

"É uma subconsulta nomeada que fica no topo da query, usando WITH. Torna o código mais legível e permite reutilizar a mesma subconsulta várias vezes. Uso muito para quebrar queries complexas em passos lógicos."

"O que é Window Function?"

"Funções que calculam sobre um conjunto de linhas relacionadas à linha atual, sem colapsar o resultado como GROUP BY faz. Exemplo: RANK() para ranking, LAG() para comparar com o valor anterior, SUM() OVER para acumulado."

"Me explique o que é RevPAR"

"Revenue Per Available Room — é a receita de quartos dividida pelo total de quartos disponíveis. É o KPI mais importante porque considera tanto o preço (ADR) quanto a ocupação. Um hotel pode ter ADR alto mas ocupação baixa, e outro o contrário — o RevPAR mostra qual realmente performa melhor."

"Qual a diferença entre Tableau e Qlik Sense?"

"Tableau é muito forte em visualização e exploração ad-hoc — a interface drag-and-drop é intuitiva e os gráficos são bonitos. Qlik Sense tem o motor associativo como diferencial — quando você clica em qualquer valor, ele filtra automaticamente todo o modelo de dados, mostrando valores associados e excluídos. Tableau trabalha mais com queries sob demanda, enquanto Qlik carrega tudo em memória."

"Como você garante que os dados do dashboard estão corretos?"

"Sempre faço validação cruzada: comparo o total do dashboard com o total do financeiro ou do sistema fonte. Se tenho 100 reservas no PMS, preciso ter 100 no BI. Também verifico outliers — se aparece uma diária de R$50.000, provavelmente é erro de digitação. E crio uma rotina de verificação automática com alertas quando os números fogem do padrão esperado."

Perguntas técnicas — ETL, Data Warehouse e Modelagem

"O que é ETL e como você implementaria aqui?"

"ETL é Extract, Transform, Load — o processo de extrair dados dos sistemas-fonte, transformar (limpar, padronizar, validar) e carregar num data warehouse. Num hotel como a Enjoy, eu extrairia do PMS as reservas e quartos, do PDV as vendas de A&B, e do ERP os dados financeiros. Na transformação, padronizaria nomes de canais, calcularia métricas como noites e diária média, e validaria outliers. Na carga, alimentaria tabelas fato e dimensão num star schema."

"Qual a diferença entre OLTP e OLAP?"

"OLTP é o banco transacional — o PMS do hotel, por exemplo — otimizado para muitas escritas rápidas como registrar reservas e check-ins. OLAP é o data warehouse — otimizado para leituras complexas, agregações e análises históricas. Nunca se faz BI direto no OLTP porque as queries pesadas de análise atrapalham a operação."

"O que é um Star Schema? Por que usar?"

"É um modelo dimensional com uma tabela fato central — contendo métricas numéricas como valor, noites, custo — conectada a dimensões ao redor — tempo, hóspede, quarto, canal. O star schema é preferido para BI porque simplifica as queries (menos JOINs), é intuitivo para ferramentas como Tableau e Qlik, e tem boa performance em agregações."

"Como você garante a qualidade dos dados no pipeline?"

"Implemento validações em cada etapa: na extração, verifico completude e volume esperado. Na transformação, trato NULLs, detecto duplicatas com ROW_NUMBER, e valido ranges (diária entre R$50 e R$5000, por exemplo). Na carga, comparo totais com a fonte e gero logs de anomalias. Também crio dashboards de data quality para monitorar essas métricas continuamente."

"O que é SCD (Slowly Changing Dimension)?"

"É como lidamos com mudanças em dados de dimensão ao longo do tempo. Tipo 1 sobrescreve — uso para correções. Tipo 2 cria nova linha com datas de vigência — uso quando preciso do histórico, como quando a tarifa de um quarto muda. Tipo 3 adiciona coluna para o valor anterior. Na hotelaria, Tipo 2 é importante para dim_quarto (mudança de tarifa) e dim_canal (mudança de comissão)."

"Você já identificou uma tendência nos dados que gerou valor?"

"Sim. Num projeto, analisei dados de vendas e identifiquei que [adapte ao seu caso: certo canal tinha ticket alto mas alta taxa de cancelamento / certos horários tinham pico de demanda desproporcional / sazonalidade diferente do esperado]. Isso levou a [ação tomada]. É o tipo de insight que a vaga pede — usar dados para identificar tendências e melhorias que agreguem valor ao negócio."

"Como você desenharia um data warehouse para a Enjoy?"

"Primeiro, definiria as camadas: bronze (dados brutos do PMS, PDV e ERP), silver (dados limpos e padronizados), e gold (star schema para consumo). O modelo teria pelo menos duas tabelas fato — fato_reservas (granularidade por reserva) e fato_receitas (granularidade por transação). As dimensões seriam tempo, hóspede, quarto, canal e departamento. A dim_tempo teria flags de feriado e temporada, crucial para análise de sazonalidade. O pipeline ETL rodaria diariamente, com validações automáticas comparando totais com as fontes."

🔵 2. Occupancy — Taxa de Ocupação

Fórmula
Quartos Vendidos ÷ Quartos Disponíveis × 100
Benchmark Resort BR
55–75%

O que mede

Percentual de quartos ocupados. Mede eficiência de venda e demanda real.

  • Ocupação alta → operação cheia, mas pode estar dando desconto para encher

🔵 3. RevPAR — Revenue per Available Room

Fórmula
ADR × Ocupação
Fórmula alternativa
Receita Quartos ÷ Quartos Disponíveis

O que mede

Performance comercial geral — combina preço E ocupação num único número. É o KPI mais citado na hotelaria.

Limitação crítica

RevPAR ignora custos. Um hotel pode ter RevPAR crescendo 10% — mas se os custos operacionais crescem 15%, o lucro caiu. Por isso o Diretor Financeiro olha GOPPAR, não só RevPAR.

🟢 4. TRevPAR — Total Revenue per Available Room

Fórmula
Receita TOTAL ÷ Quartos Disponíveis
Inclui
Quartos + A&B + Spa + Loja + Eventos

O que mede

Capacidade de monetizar o hóspede além da diária. Num resort como a Enjoy, o TRevPAR pode ser 40-60% maior que o RevPAR, porque restaurante, bar, spa e shopping geram receita extra significativa.

Insight de negócio

Hotel bom não vende só diária — vende a experiência inteira. Se o TRevPAR está muito próximo do RevPAR, significa que o hóspede não está gastando quase nada além do quarto. Isso pode indicar: (1) restaurante caro demais, (2) pouca oferta de serviços, (3) hóspedes saindo para comer fora. Num resort que tem shopping anexo como a Enjoy, o TRevPAR deveria estar bem acima.

🟢 5. GOPPAR — Gross Operating Profit per Available Room KPI do CFO

Fórmula
(Receita Total − Custos Operacionais) ÷ Quartos Disponíveis
O que realmente mede
Geração real de valor por quarto

Por que esse é O KPI para o Diretor Financeiro

RevPAR mede força comercial. GOPPAR mede se o hotel ganha dinheiro. Um hotel pode ter RevPAR crescendo enquanto o GOPPAR cai — basta os custos subirem mais que a receita.

O entrevistador compartilhou um post dizendo exatamente isso: "Revenue is activity. Profit is value creation." Se você mencionar GOPPAR naturalmente na entrevista, ele vai reconhecer que você entende a métrica que ele mais valoriza.

O que o GOPPAR captura que os outros não capturam

  • Eficiência operacional — se o custo de pessoal está alto demais
  • Escalabilidade — se crescer ocupação gera lucro ou só custo
  • Sustentabilidade — se o crescimento de receita se traduz em valor real

GráficoQuando usar

🟢 6. NRevPAR — Net Revenue per Available Room

Fórmula

Qualidade da receita. Nem toda receita é igual — R$500 via Booking (15-20% de comissão) rende R$400-425 líquidos. R$500 via site direto rende R$500.

🔴 7. CPOR — Cost per Occupied Room Conector

Fórmula
Custos Operacionais ÷ Quartos Vendidos
Inclui
Pessoal + Suprimentos + Energia + Manutenção

O que mede

Quanto custa manter cada quarto ocupado. Esse KPI conecta receita e custo — é o outro lado da equação que o GOPPAR resume.

Por que é perigoso quando sobe

  • Se CPOR ↑ rápido → cada ponto de ocupação a mais gera MENOS lucro
  • Se CPOR > ADR → cada quarto vendido dá PREJUÍZO
  • Ocupação alta pode DESTRUIR margem se CPOR não for controlado

🧩 A Pirâmide dos KPIs — Como tudo se conecta

┌──────────┐ │ GOPPAR │ ← O que o CFO quer │ lucro/ │ │ quarto │ ┌────┴────┬─────┴────┐ │TRevPAR │ CPOR │ │receita │ custo │ │total/ │ por │ │quarto │ quarto │ ┌────┴────┐ └──────────┘ │ RevPAR │ │ receita │ │ quartos/│ │ disponív│ ┌────┴──┬──────┴────┐ │ ADR │ Ocupação │ │preço │ volume │ └───────┴───────────┘

De baixo para cima:

  1. ADR × Ocupação = RevPAR → força comercial (quartos)
  2. RevPAR + receita extra = TRevPAR → monetização total do hóspede
  3. TRevPAR − CPOR ≈ GOPPAR → lucro real por quarto disponível

SQL — Dashboard executivo completo (todos KPIs juntos)

WITH room_data AS ( SELECT DATE_TRUNC('month', r.checkin) AS mes, SUM(r.valor_total) AS receita_quartos, SUM(DATE_PART('day', r.checkout - r.checkin)) AS rn_vendidas, SUM(r.valor_total * c.comissao_pct / 100) AS custo_distribuicao FROM reservas r JOIN canais c ON c.id = r.canal_id WHERE r.status = 'finalizada' GROUP BY 1 ), all_receita AS ( SELECT DATE_TRUNC('month', data) AS mes, SUM(valor) AS receita_total FROM receitas GROUP BY 1 ), all_custos AS ( SELECT DATE_TRUNC('month', data) AS mes, SUM(valor) AS custo_total FROM despesas GROUP BY 1 ), rn_disp AS ( SELECT DATE_TRUNC('month', d.data) AS mes, COUNT(d.data) * (SELECT COUNT(*) FROM quartos WHERE status = 'ativo') AS total FROM calendario d GROUP BY 1 ) SELECT rd.mes, -- Comercial ROUND(rd.receita_quartos / NULLIF(rd.rn_vendidas, 0), 2) AS adr, ROUND(rd.rn_vendidas * 100.0 / rn.total, 1) AS ocupacao, ROUND(rd.receita_quartos / rn.total, 2) AS revpar, -- Qualidade ROUND((rd.receita_quartos - rd.custo_distribuicao) / rn.total, 2) AS nrevpar, -- Operacional ROUND(ar.receita_total / rn.total, 2) AS trevpar, ROUND(ac.custo_total / NULLIF(rd.rn_vendidas, 0), 2) AS cpor, -- Resultado ROUND((ar.receita_total - ac.custo_total) / rn.total, 2) AS goppar, ROUND((ar.receita_total - ac.custo_total) * 100.0 / ar.receita_total, 1) AS gop_margin FROM room_data rd JOIN all_receita ar ON ar.mes = rd.mes JOIN all_custos ac ON ac.mes = rd.mes JOIN rn_disp rn ON rn.mes = rd.mes ORDER BY rd.mes;
